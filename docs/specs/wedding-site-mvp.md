# Одностраничный свадебный сайт MVP

## 1. Title and summary

Статический одностраничный сайт для гостей свадьбы Алисы и Александра. Сайт содержит основную информацию о событии, первичный дресс-код и RSVP-форму; реальная отправка в Telegram откладывается до появления серверного слоя.

## 2. Goals and success criteria

- Гость видит имена пары, дату, время и место проведения.
- Гость понимает общий ориентир по цветам одежды и видит места под будущие примеры образов.
- Гость может заполнить RSVP-форму отдельно за одного человека.
- MVP не раскрывает Telegram bot token и не содержит настоящей интеграции с Telegram API.
- Проект можно опубликовать на GitHub Pages через GitHub Actions без build step.

## 3. Scope / out of scope

В scope входят статические `HTML/CSS/JS`, простая адаптивная верстка, моковая отправка формы, GitHub Pages workflow и эта спецификация.

Вне scope остаются финальный визуальный дизайн, реальные фотографии/референсы образов, точные hex-цвета палитры, антиспам, хранение ответов, Telegram backend и privacy notice под формой.

## 4. Users and UX flows

Основной пользователь - приглашенный гость свадьбы.

Поток:
1. Гость открывает сайт.
2. Просматривает дату, время, место и дресс-код.
3. Переходит к анкете.
4. Заполняет имя, фамилию и статус присутствия.
5. Если выбирает `Нет`, поля алкоголя и транспорта скрываются.
6. Если выбирает `Да` или `Пока не знаю`, заполняет предпочтения в алкоголе и транспорт.
7. Нажимает отправку, видит загрузку около 1 секунды и success-сообщение.

## 5. Architecture and components

- `index.html` - структура страницы и форма.
- `styles.css` - базовая адаптивная стилизация MVP.
- `script.js` - условная логика формы, валидация и моковая отправка.
- `.github/workflows/pages.yml` - публикация статических файлов на GitHub Pages.
- `.nojekyll` - маркер, чтобы GitHub Pages отдавал статические файлы без Jekyll-обработки.

Проект не использует npm, сборщик или внешние frontend-библиотеки.

## 6. Data models and schemas

Одна отправка соответствует одному гостю.

```json
{
  "guest": {
    "firstName": "Алиса",
    "lastName": "Иванова"
  },
  "attendance": "yes",
  "transport": "transfer",
  "alcohol": {
    "options": ["wine", "sparkling"],
    "other": null
  },
  "submittedAt": "2026-05-23T13:00:00.000Z"
}
```

Значения:
- `attendance`: `yes`, `no`, `unknown`;
- `transport`: `transfer`, `self`, отсутствует при `attendance = no`;
- `alcohol.options`: `wine`, `sparkling`, `beer`, `strong`, `none`;
- `alcohol.other`: строка или `null`.

## 7. APIs and contracts

На MVP нет сетевого API. Submit формирует payload будущего контракта, показывает loading 1 секунду, затем success-состояние и пишет payload в `console.info`.

Будущая Telegram-интеграция должна идти через серверный endpoint, например Cloudflare Worker. Frontend будет отправлять `POST` с JSON payload на этот endpoint. Telegram bot token должен храниться только в server-side секретах.

## 8. Security, privacy, and compliance

- Telegram bot token запрещено хранить в `index.html`, `script.js`, GitHub Pages env или любом публичном frontend bundle.
- Форма публичная и открытая, антиспам на MVP отсутствует по продуктовой договоренности.
- Privacy notice не добавляется по продуктовой договоренности.
- Данные в MVP никуда не отправляются и не сохраняются удаленно.

## 9. Risks, tradeoffs, and mitigations

- Риск: публичная форма без антиспама при будущей реальной отправке. Митигация: добавить серверную rate limit/секретный код приглашения перед подключением Telegram.
- Риск: GitHub Pages не умеет безопасно хранить runtime-секреты. Митигация: использовать serverless backend с секретами.
- Риск: финальная палитра пока не утверждена. Митигация: использовать смысловые цвета и заменить их на финальном дизайн-этапе.

## 10. Test plan and acceptance criteria

- `index.html` открывается локально без сборки.
- На desktop и mobile нет горизонтального overflow, форма читаема.
- Имя, фамилия и присутствие обязательны.
- При `Нет` поля алкоголя и транспорта скрываются и не участвуют в payload.
- При `Да` или `Пока не знаю` транспорт обязателен, а алкоголь требует хотя бы один чекбокс или поле `Другое`.
- Submit показывает loading около 1 секунды и затем success.
- Payload появляется в консоли в ожидаемой структуре.
- GitHub Actions workflow публикует статический сайт через Pages artifact.

## 11. Rollout, observability, and rollback

Rollout: включить GitHub Pages с источником `GitHub Actions`, затем запушить `main`.

Observability MVP: ручная проверка страницы и консоли браузера.

Rollback: откатить последний commit или отключить Pages deployment в настройках репозитория.

## 12. Open questions

Открытых high-impact вопросов для MVP нет. Точные цвета, фото образов, адрес/карта и Telegram backend намеренно вынесены в следующие этапы.
